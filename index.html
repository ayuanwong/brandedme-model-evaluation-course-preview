线上指标与离线指标

理解评测，必须先理解两组指标的区别——这是评测存在的根本原因。

线上指标离线指标
定义产品上线后，用户真实行为产生的数据在部署前，用固定数据集跑出的评测分数
典型指标CTR、停留时长、转化率、用户满意度F1、BLEU、准确率、延迟
获取成本极高：需上线、等流量、跑 AB 实验低：本地或测试环境即可完成
时效性滞后：上线后才能拿到即时：模型训练完即可评测
可靠性高：反映真实用户行为取决于评测数据与标准的质量
核心矛盾：线上指标最真实但成本最高，离线指标成本低但可能不准。评测的本质就是建立一套可靠的离线指标体系，使其尽可能逼近线上指标，从而在低成本下做出正确决策。

一、为什么需要模型评测

不做评测会怎样？评测对哪些环节"做功"？

一个真实案例

某电商平台上线了一个客服大模型，内部测试准确率 96%。上线两周后投诉量激增 40%。排查发现：

  • 测试集 70% 是"查物流"这类简单问题，长尾场景（退款/投诉）几乎没覆盖
  • 被引导时，12% 的回复包含未验证的退款承诺
  • 不同地区用户的解决率差异达 25%，但均值掩盖了这一偏差
核心问题：不是模型不够好，而是评测没有覆盖真实场景的风险。96% 的准确率是假象——离线指标和线上真实表现之间存在巨大鸿沟。

评测的目的：对哪些环节"做功"

环节评测的作用离线→线上逻辑
需求定义将"什么算好"从模糊的业务语言转化为可量化的评测标准评测标准 = 业务方与算法团队的"合同"
模型选型用离线指标对比候选模型，替代昂贵的线上AB测试离线指标可靠的选型 → 线上表现符合预期
上线决策用离线指标判断是否达到可部署标准离线指标达标 → 线上风险可控
迭代方向定位薄弱环节，指导优化优先级离线指标分层分析 → 发现线上会出问题的子场景
风险防控发现安全漏洞、公平性偏差等隐性风险离线专项测试 → 覆盖线上极端场景
一句话总结：评测的本质是缩小离线指标与线上指标之间的鸿沟——让产品决策从经验驱动升级为数据驱动。

二、评测的核心概念与方法

评什么？怎么评？维度之间如何权衡？

三种评测类型

类型关注点典型场景回答的问题
基准评测标准化数据集上的表现在 MMLU/GSM8K 上跑分"这个模型能干什么？"
能力评测特定任务上的实用性测试客服对话、代码生成能力"在我的场景里好用吗？"
对齐评测模型输出是否符合预期约束检测有害输出、偏见、风格偏移、拒绝能力"输出是不是我们想要的？"

对齐评测不止是"安全"

很多人把"对齐"等同于"安全与价值观"，这是狭隘的理解。对齐评测的核心是验证模型输出是否符合预期约束，包括但不限于：

  • 安全对齐：不输出有害内容、不泄露隐私（传统理解）
  • 风格对齐：客服模型是否始终礼貌、简洁？写作助手是否保持了指定语气？
  • 角色对齐：模型是否"扮演"了正确的角色？是否有越界行为？
  • 格式对齐：输出是否符合要求的格式（JSON/Markdown/结构化表格）？

举例：一个客服模型即使 100% 安全，但如果说话风格像"学术论文"而非"亲切客服"，用户同样不会满意——这就是风格对齐的问题。

能力评测：对标真人

能力评测最有效的思路是对标真人——把人的能力项拆解出来，逐一评测模型是否达标。

PM 的核心价值：找到业务最优的权衡点

  • 准确性 vs 延迟：大模型更准但更慢，业务对速度的容忍度决定了上限
  • 准确性 vs 成本：高性能 = 高算力 = 高成本，免费产品对成本极度敏感
  • 安全性 vs 能力：过度保守可能拒绝合理请求，降低实用性

没有"万能模型"，只有在特定业务约束下的最优权衡。

三、评测体系设计与实施

先捋清指标体系，再分层定义指标，最后落地执行

评测体系设计的第一步：捋清指标体系

很多人直接跳到"用什么指标"，这是错的。第一步是建立指标体系的整体框架——先有骨架，再填血肉。

指标体系的分层结构

点击每一层查看详细说明——从顶层到底层，逐层细化
红线层一票否决，触碰即不通过

红线指标 = 绝对不可逾越的底线，相当于 HR 考察中的"合规/诚信"。

典型红线指标：有害输出率 > 0.5%、隐私泄露率 > 0%、特定群体歧视率 > 1%

特征：不需要优化，只需要守住；一旦越线，不管其他指标多好都不准上线。

对齐层输出是否符合预期约束

对齐指标 = 模型输出的"行为规范"，相当于 HR 考察中的"价值观/文化匹配"。

对齐维度示例：

  • 安全对齐：有害输出率、拒绝能力
  • 风格对齐：语气一致性、礼貌程度
  • 角色对齐：是否越界（如客服模型给出医疗建议）
  • 格式对齐：输出是否符合结构化要求

特征：通常是"一票否决"或"降权"——对齐不达标则核心能力分数打折。

能力层核心任务的表现

能力指标 = 模型"能不能干好活"，相当于 HR 考察中的"核心胜任力"——这是北极星指标所在层。

对标真人拆解能力项：

  • 客服模型 → 意图理解、问题解决、情感安抚、知识调用
  • 写作助手 → 内容质量、逻辑连贯、创意水平、指令遵循
  • 代码助手 → 代码正确性、可读性、边界处理、注释质量

特征：每个能力项需要独立评测 + 打分，最终加权汇总；北极星指标从这层选取。

性能层非功能性约束

性能指标 = 模型的"体力和效率"，相当于 HR 考察中的"专业硬技能"。

典型性能指标：首 Token 延迟、吞吐量、单次推理成本、并发支持量

特征：作为护栏指标，不低于业务要求的最低线即可，不需要极致优化。

设计原则：从上往下设计（先定红线→再定对齐→再定能力→最后定性能），从下往上达标（先守住性能→再提升能力→再保证对齐→最后不碰红线）。

六步闭环

  • 定义目标 -- 评什么？为什么评？谁看结果？什么算"通过"？
  • 选择指标 -- 按分层框架选择：红线→对齐→能力（北极星）→性能（护栏）
  • 构建数据集 -- 代表性、无偏性、可复现。警惕数据泄露
  • 执行评测 -- 自动化评测、人工评测、LLM-as-Judge 三种方式各有适用场景
  • 分析结果 -- 不只看均值：关注分布、长尾、Case 分析、统计显著性
  • 形成决策 -- 上线 / 暂缓 / 继续优化。建立评测基线供后续对比

实战：为"内容审核模型"设计指标体系

短视频平台的内容审核模型。按照分层框架，逐步设计评测标准。

按分层设计评测指标

红线层：什么是绝对不可接受的？
涉政涉黄内容漏过率 > 0%（一票否决）
不需要红线，F1 够高就行
对齐层：模型应该遵循什么行为约束？
安全对齐：有害输出率 ≤ 0.5%
公平对齐：不同群体误杀率差异 ≤ 5%
对抗防御：对抗样本攻击成功率 ≤ 2%
能力层：核心任务用什么指标衡量？
召回率为北极星（漏过违规代价远大于误杀）
精准率为北极星（减少误杀，降低人工复审量）
F1 为北极星（精准与召回平衡）
性能层：非功能性约束是什么？
审核延迟 ≤ 500ms
单次审核成本 ≤ 0.001元
新违规类型发现覆盖率 ≥ 80%

评测结果如何驱动迭代

评测结论后续行动
红线越线拒绝上线，优先修复红线问题
对齐不达标暂缓上线，调整模型行为约束
能力指标达标，但特定场景薄弱针对性补充数据，定向优化
能力指标全达标，护栏指标越线拒绝上线，优先解决护栏问题
离线指标提升，但线上反馈变差检查评测数据是否与线上分布一致——"尺子"可能弯了
关键原则：评测标准本身也需要迭代。如果离线指标和线上指标持续不一致，说明你的"尺子"需要校准了。

四、问题溯源与定位

评测不达标时，是评测的问题还是模型的问题？

三层排查法

评测不达标时，问题可能出在三个层面。PM 的关键是判断该动哪个——直接优化模型往往是最差的选择。

核心原则：先确认"尺子是直的"，再判断"东西是不是歪的"。

选型决策的核心逻辑

选型决策不是"谁分数高选谁"，而是基于指标体系逐层过滤：

  • 第一步：红线过滤 — 有害输出率 > 0.5% 的模型直接淘汰，不管其他指标多好
  • 第二步：对齐筛选 — 风格/角色/格式不符合业务要求的降权或淘汰
  • 第三步：能力排序 — 在红线和对齐达标的模型中，按北极星指标排序
  • 第四步：性能权衡 — 最终候选模型之间做延迟/成本/覆盖率的权衡

这个"逐层漏斗"确保了安全优先、对齐其次、能力为主、性能兜底的决策逻辑。

六、能力增强与机制设计

评测不是终点，而是驱动模型/Agent 能力持续增强的引擎

评测驱动的能力增强闭环

很多团队把评测当作"上线前的检查项"，做完就结束了。但评测真正的价值在于驱动能力迭代——评测发现问题 → 定位差距 → 优化方案 → 重新评测 → 验证提升。这是一个持续运转的飞轮。

核心逻辑：评测不是"考试"，而是"体检"——考试考完就结束了，体检是为了找到问题并给出治疗方案。评测体系的质量，直接决定了能力增强的方向是否正确。

完整的体验流程：从业务需求到能力增强

以下是一个 AI 产品从"发现能力不足"到"能力增强落地"的完整链路。每一步都有评测参与。

阶段做什么评测的角色关键产出
1. 发现问题线上指标下降 / 用户投诉 / 新场景需求评测体系作为"监测器"，及时发现问题信号问题清单（按优先级排列）
2. 定位差距用评测体系拆解：是哪个能力项不足？差距多大？分层评测 → 精确定位薄弱能力项差距分析报告
3. 确定方案判断差距类型：模型问题 / 数据问题 / Prompt 问题 / 系统问题评测提供诊断依据——同数据不同模型、同模型不同数据等对比实验优化方案（可能不只改模型）
4. 执行优化微调 / 补数据 / 调 Prompt / 加规则 / 换模型——（此步评测不直接参与，但优化目标由评测定义）优化后的模型/系统
5. 重新评测用同一套评测体系重新跑分，验证是否提升同一标准下的前后对比 → 量化提升幅度优化前后对比报告
6. 线上验证AB 实验 / 灰度发布，验证线上指标是否改善确认离线指标提升是否传导到线上——如果不一致，回溯校准评测体系上线决策 + 评测体系校准记录

Agent 的评测维度

Agent 由多个能力模块协作组成，评测时需要拆开看每个模块 + 合起来看整体：

  • 意图理解能力 — 用户到底要什么？理解错了后面全错
  • 工具调用能力 — 选对工具 + 参数正确 + 处理返回结果

评测体系本身的迭代

能力增强飞轮持续运转的过程中，评测体系本身也需要迭代。什么时候需要迭代评测体系？

  • 离线指标持续提升，但线上指标不变 → 评测体系没有反映真实的业务目标，需要校准
  • 新场景/新能力上线 → 评测体系需要扩展覆盖范围
  • 线上出现了评测未覆盖的问题 → 评测体系有盲区，需要补充
  • 评测结果的区分度下降（所有模型都接近满分）→ 评测太简单了，需要提高难度

一句话：评测体系不是一次性的，它和模型一样需要持续迭代。模型是"学生"，评测是"考卷"——考卷太简单，学生考满分也不代表真会了。
